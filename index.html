Description Writing Guide ✕

Goal

Write a plain-language description of your business process. The AI will convert it into a diagram with two tables: Elements (stages) and Connections (arrows between stages).

What to include

  • Who performs each stage (person or role name)
  • What they do at that stage
  • What button they click to move to the next stage
  • Where the flow goes after each action
  • Conditions — if the system routes differently based on data (e.g. "if approver is senior manager…")
  • Backward paths — if a stage can return to a previous one (e.g. "if rejected, goes back to…")
  • The final state — what the closed/completed form looks like and who sees it

How each element is understood

  • User stage — a person actively works on a form and clicks a button to proceed. Must have a named user and a button.
  • System check — the system automatically evaluates a condition and routes to one of two or more paths. Not a user decision.
  • System action — the system automatically does something in the background (assign, notify, calculate) with no user involvement.
  • Final closed state — the last user stage before the process ends. It is still a user stage, not the end itself.

What NOT to include

  • Form field names or data structures
  • UI layout or visual details
  • Notifications or emails (unless they trigger a stage transition)
  • Internal calculations that don't affect the flow
  • Technical system names or database terms

Tips

  • One sentence per action is enough
  • Name every person/role — "the manager", "the operator", "the approver"
  • Name every button — "clicks Send", "clicks Approve", "clicks Reject"
  • If something is unclear, use the Correction Prompt to let the AI fill gaps with placeholders first
